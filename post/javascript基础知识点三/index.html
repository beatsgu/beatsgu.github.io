JavaScript基础知识点（三）：闭包解析与作用域讲解

1. 前言

闭包是 JavaScript 中最重要也是最容易混淆的概念之一。它不是某种特殊的语法，而是作用域规则的自然结果。本文将从基础概念开始，深入探讨闭包的形成机制、经典应用场景，以及相关的作用域知识。

CSDN文章链接

2. 什么是闭包

2.1 闭包的定义

闭包的定义很简单：函数 A 内部有一个函数 B，函数 B 可以访问到函数 A 中的变量，那么函数 B 就是闭包。

在 JavaScript 中，闭包存在的意义就是让我们可以间接访问函数内部的变量。

重要提示：闭包不是"语法特性"，而是"作用域规则的自然结果"。

2.2 闭包的形成条件

闭包需要满足三个条件：

  1. 嵌套函数：函数内部定义另一个函数
  2. 访问外部变量：内部函数访问外部函数的变量
  3. 外部引用：内部函数被外部作用域引用（通常是被返回或传递）

3. 闭包的基本形成机制

3.1 经典闭包示例

最经典的闭包场景是"函数嵌套 + 返回内层函数"：

// 1. 定义外层函数 outer，内部有局部变量 num
function outer() {
  let num = 10; // 外层函数的局部变量（本应只在 outer 内部访问）
  
  // 2. 定义内层函数 inner（闭包的核心载体）
  function inner() {
    console.log(num); // 内层函数访问外层的 num（触发作用域链查找）
  }
  
  // 3. 关键：将内层函数 inner 作为返回值"带出"外层作用域
  return inner;
}

// 4. 执行 outer()，得到返回的 inner 函数，赋值给全局变量 fn
const fn = outer(); 

// 5. 此时 fn（即 inner）已在全局作用域执行，但依然能访问 outer 的 num！
fn(); // 输出：10 → 这就是闭包的效果

3.2 关键疑问：为什么 outer 执行完，num 没被销毁？

正常情况下，函数执行完后，其局部作用域会被 JavaScript 的"垃圾回收机制（GC）“销毁（变量也会被回收）。

但闭包中，因为 inner 函数"引用"了 num，且 inner 被带出了 outer 作用域（存在于全局），GC 会认为 num 仍有用，因此保留外层函数的作用域——这就是闭包能"记住"外层变量的本质。

4. 闭包的判断标准

4.1 关键判断原则

闭包的核心不是"内部函数是否被返回”，而是：

“内部函数保留了对外部函数作用域的引用，且该内部函数能在外部函数作用域之外被调用”

判断是否形成闭包，关键看**“内部函数是否能在外部函数作用域之外被调用”**。

4.2 情况分析

4.2.1 情况1：内部函数仅在外部函数内部调用 → 不算闭包

如果内部函数既没被返回，也没被赋值给外部函数外的变量/对象，仅在外部函数内部执行，那么它本质只是"普通的嵌套函数"，不算闭包。

function outer() {
  let num = 10;
  
  // 内部函数 inner 仅在 outer 内部调用
  function inner() {
    console.log(num); // 访问外层变量，但仅在 outer 内部执行
  }
  
  inner(); // 只在 outer 内部调用，未"带出"外部
}

outer(); // 执行后，outer 的作用域正常销毁（num 随之消失）

此时 inner 虽然访问了 num，但它的调用完全局限在 outer 内部，没有脱离 outer 的作用域 → 不形成闭包。

4.2.2 情况2：内部函数被"间接带出"外部 → 算闭包

如果内部函数虽然没被直接返回，但被赋值给了外部函数外的变量/全局对象，那么它仍能在外部函数作用域之外被调用，此时依然会形成闭包。

let globalFunc; // 外部全局变量

function outer() {
  let num = 10;
  
  function inner() {
    console.log(num); // 访问外层变量
  }
  
  // 关键：不直接返回 inner，而是将其赋值给外部的 globalFunc
  globalFunc = inner; 
}

// 1. 先执行 outer：此时 inner 被赋值给 globalFunc（带出外部）
outer(); 

// 2. 执行 globalFunc（本质是执行 inner）：此时 outer 已执行完毕，但 num 仍能被访问
globalFunc(); // 输出 10 → 形成闭包

这里 inner 没被直接返回，但通过赋值给外部的 globalFunc，成功"脱离"了 outer 的作用域 → 形成闭包。

5. 闭包经典案例：循环中的 setTimeout

5.1 问题场景 

for (var i = 1; i <= 5; i++) {
  setTimeout(function timer() {
    console.log(i);
  }, i * 1000);
}

// 预期输出：1, 2, 3, 4, 5
// 实际输出：6, 6, 6, 6, 6

5.2 问题分析

由于 var 声明的 i 是函数作用域，并且 setTimeout 的回调是异步执行的：

  1. 循环瞬间完成，i 变成 6
  2. 1秒后，所有 setTimeout 回调开始执行
  3. 但此时 i 已经是 6，所以输出一堆 6

5.3 闭包解决方案

5.3.1 使用立即执行函数（IIFE）创建闭包 

for (var i = 1; i <= 5; i++) {
  (function(j) {
    // j 是当前循环中 i 的副本
    setTimeout(function timer() {
      console.log(j);
    }, j * 1000);
  })(i); // 立即执行，传入当前的 i
}

// 输出结果：1, 2, 3, 4, 5

原理分析：

  • 每次循环都调用一个立即执行函数，j 是形参，保存了当前 i 的值
  • setTimeout 回调引用的是 j，而每个 j 都在不同的闭包中，互不干扰

5.3.2 补充讲解

一、理解 (function(j) { ... })(i)

这个可以拆开来看

1. (function(j) { ... })

这是一个匿名函数，它被括号 () 包裹起来。

function(j) {
  setTimeout(function timer() {
    console.log(j)
  }, j * 1000)
}

这只是一个函数定义，不会立即执行。就像你写：

function sayHello() { console.log("Hello") }

它只是定义，没调用。

但加上括号 ( ) 后：

(function(j) { ... })

这只是为了让 JavaScript 知道：“这是一个函数表达式，不是函数声明”，为下一步调用做准备。

2. (function(j) { ... })(i) —— 立即调用！

后面的 (i) 是在调用这个函数，并传入当前的 i 值。

所以：

(function(j) { ... })(i)

等价于：

var tempFunction = function(j) {
  setTimeout(function timer() {
    console.log(j)
  }, j * 1000)
}
tempFunction(i)  // 立即执行，传入 i

这种写法叫做 IIFE（Immediately Invoked Function Expression），中文叫：立即调用函数表达式。

二、如何形成的闭包？

代码：

(function(j) {
  setTimeout(function timer() {
    console.log(j)  // <--- 这里引用了 j
  }, j * 1000)
})(i)
  • 外层的 function(j) 被立即执行，j 接收了当前 i 的值（比如第一次是 1）。
  • 然后它创建了一个 setTimeout，延迟执行一个函数 timer。
  • timer 函数中引用了变量 j。
  • 当 timer 几秒后执行时，外层的 function(j) 早就执行完了，但 timer 依然能访问到 j。

这就是闭包的本质：内部函数（timer）捕获了外部函数（function(j)）的变量（j），即使外部函数已经执行完毕。

5.4 使用 let

除了使用闭包还有一种更简单的方法，那就是使用 let

for (let i = 1; i <= 5; i++) {
  setTimeout(function timer() {
    console.log(i);
  }, i * 1000);
}

// 输出结果：1, 2, 3, 4, 5

原理：let 是块级作用域，每次循环都会创建一个新的 i，每个 setTimeout 回调都绑定到对应轮次的 i，本质上也形成了闭包。

5.4.1补充讲解：

这个使用let形成块级作用域是如何解决的？

原因：let 在 for 循环中每次迭代都会创建一个新的块级作用域变量

1.普通的 let 块级作用域（基础）

{
  let x = 1
}
console.log(x) // 报错：x is not defined

这说明 let 是块级作用域，出了 {} 就访问不到。

2.但在 for 循环中，let 的行为更“聪明”

for (let i = 1; i <= 5; i++) {
  setTimeout(() => console.log(i), i * 1000)
}

JavaScript 引擎在背后实际上是这样处理的：

// 第1次循环
let i1 = 1
setTimeout(() => console.log(i1), 1000)

// 第2次循环
let i2 = 2
setTimeout(() => console.log(i2), 2000)

// 第3次循环
let i3 = 3
setTimeout(() => console.log(i3), 3000)

// ...

每次循环，let i 都创建了一个新的变量实例，而不是复用同一个 i。

3.为什么这能解决 setTimeout 的问题？

  • setTimeout 的回调函数引用的是当前轮次的 i。
  • 因为每次循环的 i 都是不同的变量（在不同的词法环境中），所以每个回调都绑定到自己的 i。
  • 即使循环结束，这些 i 仍然被各自的回调函数“捕获”——这其实就是闭包！

所以：let 在 for 循环中自动形成了闭包。

6.var的函数作用域和let的块级作用域

6.1 函数作用域 vs 块级作用域

6.1.1 什么是函数作用域（var）

使用 var 声明的变量：

  • 只在函数内部有作用域
  • 在函数内的任何地方（包括 if、for、while 等块中）声明，都会被提升到整个函数的顶部
  • 在函数外不可访问

6.1.2 什么是块级作用域（let/const）

在 JavaScript 中，“块” 是指用大括号 {} 包裹的一段代码，比如：

  • if 语句
  • for 循环
  • while 循环
  • 普通的 {} 代码块

使用 let/const 声明的变量： 只能在声明它的那个 {} 块内访问，出了这个块就"消失"了。

6.2 作用域对比案例 

function scopeTest() {
  console.log("函数开始");
  
  if (true) {
    var varInBlock = "var在块中";
    let letInBlock = "let在块中";
    const constInBlock = "const在块中";
    
    console.log(varInBlock);   // "var在块中"
    console.log(letInBlock);   // "let在块中"
    console.log(constInBlock); // "const在块中"
  }
  
  console.log(varInBlock);   // "var在块中" (var 没有块级作用域)
  // console.log(letInBlock);   // ReferenceError
  // console.log(constInBlock); // ReferenceError
  
  // switch 语句中的块级作用域
  switch (1) {
    case 1:
      let caseVar = "case变量";
      break;
    // case 2:
    //   let caseVar = "另一个case变量"; // 重复声明
    //   break;
  }
}

6.3 为什么 var 会导致 setTimeout 问题？ 

// var 的函数作用域特性
for (var i = 0; i < 3; i++) {
  // var i 在整个函数作用域中只有一个
  // 等价于：
  // var i;
  // for (i = 0; i < 3; i++) { ... }
}
console.log(i); // 3 (在循环外仍可访问)

// let 的块级作用域特性
for (let j = 0; j < 3; j++) {
  // 每次循环都创建一个新的 j
}
// console.log(j); // ReferenceError (块外无法访问)

7.闭包是否会造成内存泄漏？

回答：闭包本身不一定会造成内存泄漏，但在某些情况下确实可能导致内存泄漏问题。

7.1 闭包的内存机制

闭包会保持对外部作用域变量的引用，这意味着这些变量不会被垃圾回收器回收，即使外部函数已经执行完毕。这是闭包的正常工作机制，不算是内存泄漏。

7.2 导致内存泄漏情况

  1. 持有本该被销毁的函数，造成其关联的词法环境无法销毁。

    function createXXX() {
    const big = new Array(7000000).fill('x'); // 大型数组

    function s1() {
        big; // s1 引用了 big
    }

    return s1; // 返回了 s1 函数
}

const xxx = createXXX(); // xxx 持有 s1 的引用
// 即使 createXXX 执行完毕，big 也无法被回收，因为 s1 还在引用它

    问题分析：

    • s1 函数形成闭包，引用了外部的 big 变量
    • 外部代码持有 s1 的引用（通过 xxx）
    • 导致包含 big 的整个词法环境无法被销毁
    • big 数组（约28MB内存）一直占用内存

  2. 当有多个函数共享此法环境时，可能导致词法环境膨胀，从而发生无法访问但无法销毁的数据。

    function createXXX() {
    const big = new Array(7000000).fill('x');   // 大数据
    const small = 'x';                          // 小数据

    function s1() {
        big;    // s1 引用 big
    }

    function s2() {
        small;  // s2 引用 small
        return s2;
    }

    return s2; // 只返回 s2
}

    问题分析：

    • s1 和 s2 都在同一个词法环境中定义
    • 它们共享同一个词法环境，该环境包含 big 和 small
    • 即使只返回 s2，但由于 s1 的存在，整个词法环境必须保持
    • 导致 big 虽然无法通过返回的函数访问，但仍然无法被回收

7.3 内存泄漏的判断原则

不构成内存泄漏的情况：

  • 小数据量：基本数据类型、短字符串、小对象
  • 预期行为：闭包正常保持对外部变量的引用
  • 可控影响：即使累积也不会显著影响系统性能

构成内存泄漏的情况：

  • 大数据量：大数组、大对象、多媒体数据
  • 持续增长：随时间推移内存占用不断增加
  • 无法释放：应该被回收但无法回收的内存
  • 影响性能：导致页面卡顿、崩溃等问题

8. 一句话总结

闭包是"内层函数携带外层变量引用"的特殊函数，它打破了"外层不能访问内层变量"的限制，可用于实现私有变量、缓存数据，但需注意内存泄漏风险。

9. 小结

本文深入探讨了 JavaScript 闭包的核心概念：

  1. 闭包定义：内部函数访问外部函数变量，且能在外部作用域调用
  2. 形成条件：嵌套函数 + 访问外部变量 + 外部引用
  3. 经典应用：解决循环中的异步问题（setTimeout）
  4. 作用域区别：var（函数作用域）vs let/const（块级作用域）
  5. 注意事项：避免内存泄漏
