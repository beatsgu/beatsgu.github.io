Java并发编程

进程

程序是静态的，进程可以看作是程序的一个实例。

二者对比

  • 进程是相互独立的，线程存在于进程内，是进程的一个子集
  • 进程拥有共享的资源，如内存空间，供其内部的线程共享。
  • 进程间通信较为复杂
    • 同一台计算机的进程通信称为 IPC
    • 不同计算机之间的进程通信，需要通过网络，并遵守共同的协议，例如HTTP
  • 线程通信相对简单，因为它们共享进程内的内存。
  • 线程更轻量，线程上下文切换成本一般上要比进程上下文切换低。

  1. ​ 单核 cpu 下，多线程不能实际提高程序运行效率，只是为了能够在不同的任务之间切换，不同线程轮流使用 cpu ，不至于一个线程总占用 cpu，别的线程没法干活
  2. ​ 多核 cpu 可以并行跑多个线程，但能否提高程序运行效率还是要分情况的 有些任务，经过精心设计，将任务拆分，并行执行，当然可以提高程序的运行效率。但不是所有计算任 务都能拆分（参考后文的【阿姆达尔定律】） 也不是所有任务都需要拆分，任务的目的如果不同，谈拆分和效率没啥意义
  3. ​ IO 操作不占用 cpu，只是我们一般拷贝文件使用的是【阻塞 IO】，这时相当于线程虽然不用 cpu，但需要一 直等待 IO 结束，没能充分利用线程。所以才有后面的【非阻塞 IO】和【异步 IO】优化

创建和运行线程

方法一 直接使用Thread 

Thread t = new Thread() {
    @Override
    public void run() {
        // 要执行的任务
    }
}
// 启动线程
t.start()

方法二 使用Runnable配合Thread

把线程和任务分开

  • Thread代表线程
  • Runnable代表可运行的任务（线程要执行的代码）
Runnable runnable = new Runnable() {
 public void run(){
 // 要执行的任务
    }
 };
 // 创建线程对象
Thread t = new Thread( runnable );
 // 启动线程
t.start();

Java8 之后可以使用lambda精简代码

// 创建任务对象
Runnable task2 = () -> log.debug("hello");
 // 参数1 是任务对象; 参数2 是线程名字，推荐
Thread t2 = new Thread(task2, "t2");
 t2.start()

追踪源码可以发现，方法一相当于直接重写了父类的run()方法，而方法二把runnable传入到Thread中，父类在调用run方法时，如果检测到有Runnable对象传入，则去执行子类的run方法。

​ 方法1 是把线程和任务合并在了一起，方法2 是把线程和任务分开了 用 Runnable 更容易与线程池等高级 API 配合 用 Runnable 让任务类脱离了 Thread 继承体系，更灵活

推荐使用方法二 组合优于继承

原理之线程运行

栈与栈帧

Java Virtual Machine Stacks（Java虚拟机栈）

JVM中由堆，栈，方法区所组成，其中栈内存就是给线程使用的，每个线程启动后，虚拟机就会为其分配一块栈内存。

  • 每个栈由多个栈帧组成，对应着每次方法调用时所占用的内存
  • 每个线程只能有一个活动栈帧，对应着当前正在执行的那个方法。

线程调用放到栈当中，线程中执行方法时再开一个对应的栈帧，方法结束返回下一条语句的地址，进行相应代码的执行。

栈和栈帧都分别存放哪些数据？

在Java虚拟机（JVM）中，栈（Java Stack）是线程私有的内存区域，用于存储方法调用相关的数据。每个线程在创建时都会分配一个独立的栈，栈中存放的数据结构是栈帧（Stack Frame），每个栈帧对应一个方法调用。以下是详细说明：

一、Java栈中存放的数据

Java栈中存放的是线程执行方法时产生的栈帧。每个栈帧代表一个方法调用，包括以下特点：

  1. 线程私有：不同线程的栈彼此隔离，不会共享。
  2. 后进先出（LIFO）：方法调用的顺序决定了栈帧的入栈（push）和出栈（pop）顺序。
  3. 动态扩展：栈的大小可以动态调整（但受-Xss参数限制，超出会抛出StackOverflowError）。

二、栈帧中存放的数据

每个栈帧包含以下核心数据结构：

  1. 局部变量表（Local Variable Array）

  • 存储内容：方法的局部变量（包括方法参数和方法内部定义的变量）。

数据类型：

  • 基本数据类型（int、long等）。

  • 对象引用（指向堆中对象的地址，如String、List等）。

  • returnAddress（指向字节码指令的地址，已很少使用）。

  • 空间分配：在编译期确定大小，运行期不会改变。

  1. 操作数栈（Operand Stack）
  • 作用：执行字节码指令的临时工作区（类似CPU的寄存器）。
  • 操作示例：如计算 a = b + c 时，b 和 c 的值会被压入操作数栈，执行加法指令后结果存入栈顶，再赋值给变量 a。
  • 空间分配：编译期确定最大深度。
  1. 动态链接（Dynamic Linking）
  • 作用：将符号引用（如com.example.MyClass#method）转换为实际内存地址的直接引用。
  • 关联数据：指向运行时常量池（Runtime Constant Pool）中该方法的引用，用于支持多态（如方法重写）。

详解动态链接

动态链接的作用

动态链接的核心是：在程序运行时，将方法的符号引用（Symbolic Reference）转换为直接引用（Direct Reference）。

  • 符号引用：可以理解为方法的“名字”，比如com.example.MyClass#calculate()。
  • 直接引用：方法在内存中的实际地址（指针），比如0x7f3e8a1b。

为什么需要动态链接？

Java支持多态（Polymorphism），即父类引用可以指向子类对象。例如：

class Animal {
    void sound() { System.out.println("Animal makes sound"); }
}
class Dog extends Animal {
    @Override
    void sound() { System.out.println("Dog barks"); }
}

public class Test {
    public static void main(String[] args) {
        Animal animal = new Dog();  // 父类引用指向子类对象
        animal.sound();  // 调用的是Dog的sound()方法，而不是Animal的！
    }
}
  • 在编译期，animal.sound()只是一个符号引用（Animal#sound()）。
  • 在运行时，JVM需要根据实际对象类型（Dog）找到真正要执行的Dog#sound()方法的内存地址。 ​​动态链接就是完成这个“找到实际方法地址”的过程​​。

动态链接如何工作？

  1. 符号引用：编译后的字节码中，方法调用（如animal.sound()）会记录为一个符号引用，包含类名、方法名和参数类型（例如Animal.sound()V）。
  2. 运行时常量池：符号引用存储在类的运行时常量池（Runtime Constant Pool）中。
  3. 动态链接：在方法调用时，JVM根据符号引用去运行时常量池查找，再结合实际对象类型（多态），找到方法的直接引用（内存地址）。
  4. 绑定方法地址：将直接引用替换到栈帧中，确保后续指令能正确执行。

4. 动态链接 vs 静态链接

  • 静态链接：在编译期就能确定方法地址（如private方法、static方法、构造函数等）。
  • 动态链接：在运行期才能确定方法地址（如重写方法、接口方法等）。

例如：

class Example {
    static void staticMethod() {}  // 静态方法（静态链接）
    void instanceMethod() {}       // 实例方法（可能动态链接）
}

public class Main {
    public static void main(String[] args) {
        Example.staticMethod();    // 静态链接：编译期确定地址
        new Example().instanceMethod(); // 动态链接：运行时确定地址
    }
}
  1. 方法返回地址（Return Address）

  • 作用：方法正常退出或异常退出后，需要返回到方法调用者的位置。

存储内容：

  • 正常返回：调用者的程序计数器（PC）值，指示下一条指令的地址。
  • 异常退出：异常处理表的地址（通过异常处理器查找）。
  1. 附加信息（可选）
  • JVM实现可能添加的额外数据，例如调试信息、性能监控数据等。

join可以理解为JS中的await，有时候我们需要等某个程序运行完才执行之后的程序，即回调

static int r = 0;
 public static void main(String[] args) throws InterruptedException {
 test1();
 }

Interrupt

打断sleep，wait，join的线程

这几个方法都会让线程进入阻塞状态。

打断sleep线程，会清空打断状态，以sleep为例。

如果是打断正常线程（不是由于sleep，wait，join等阻塞的线程），那么不会清空打断状态，isInterrupted()的值为真。

两阶段终止模式

如何在一个线程T1中“优雅”地终止T2线程？这里的【优雅】指的是给T2一个料理后事的机会。

class TwoPhaseTermination {
    private Thread monitor;  // 监控线程

    // 启动监控线程
    public void start() {
        monitor = new Thread(() -> {
            while (true) {
                Thread current = Thread.currentThread();
                // 阶段1：检查是否被中断
                if (current.isInterrupted()) { 
                    System.out.println("料理后事");  // 执行终止前的清理工作
                    break;  // 退出循环，线程自然终止
                }
                
                // 阶段2：正常执行任务（未中断时）
                try {
                    Thread.sleep(1000);          // 情况1：睡眠时被中断会抛异常
                    System.out.println("执行监控记录"); // 情况2：正常执行任务
                } catch (InterruptedException e) {
                    e.printStackTrace();
                    // 关键！重新设置中断标记，让下一次循环能检测到中断
                    current.interrupt();  
                }
            }
        });
        monitor.start();
    }

    // 外部调用此方法通知线程终止
    public void stop() {
        monitor.interrupt();  // 发出中断信号
    }
}

两阶段终止模式，关键就是可以让被打断的线程有一个“释放资源”的操作。这个模式实现的时候一个注意点就是如果被打断的线程是在sleep，wait等方法被打断的，那么不会改变被打断的信号，所以我们要手动设置一下。

interrupt了线程，线程为什么还能执行？

  • interrupt() 的作用：仅仅是给线程设置一个中断标记（类似打一个“终止请求”的标签），并不会直接终止线程。
  • 线程的主动权：线程是否终止，完全由线程自身决定。线程需要主动检查中断标记，并选择如何响应（比如退出循环、释放资源）。

stop才是强制终止线程。

在 Java 并发编程中，LockSupport.park() 是一个底层线程阻塞工具，它与 LockSupport.unpark(Thread) 配合使用，用于精准控制线程的挂起和唤醒。以下是其核心原理和关键细节：

1. park() 的核心机制

  • 许可证（Permit）模型：

    • 每个线程默认有 0 个许可证。

    • unpark(thread) 为线程添加 1 个许可证（最多累积 1 个）。

    • park()

      尝试消耗 1 个许可证：

      • 有许可证 → 立即返回（不阻塞）。
      • 无许可证 → 阻塞线程直到许可证可用或中断。

  • 非重入性：多次调用 unpark() 不会叠加许可证（始终保持最多 1 个）。

2. 基础使用示例 

Thread worker = new Thread(() -> {
    System.out.println("Worker 开始工作");
    LockSupport.park(); // 阻塞，等待许可证
    System.out.println("Worker 被唤醒");
});

worker.start();
Thread.sleep(1000);
LockSupport.unpark(worker); // 发送许可证唤醒线程

3. park() 的四大特性

(1) 精准唤醒

  • 直接指定要唤醒的线程（unpark(thread)），无需通过锁对象。

(2) 无锁要求

  • 不需要在同步代码块中调用，避免死锁风险。

(3) 中断响应

  • 若线程在

    park()

    时被中断，会立即返回并

    保留中断状态

    LockSupport.park();
if (Thread.interrupted()) {
    System.out.println("线程被中断");
}

(4) 超时支持

  • parkNanos(long)：阻塞指定纳秒后自动唤醒。
  • parkUntil(long)：阻塞到指定时间戳（毫秒）。

CAS（Compare-And-Swap）操作详解

1. CAS 是什么？

  • 定义：Compare-And-Swap（比较并交换）是一种原子操作，用于在多线程环境下实现无锁（Lock-Free）同步。
  • 核心思想： ​​“我认为内存中的值应该是A，如果是，则将其修改为B；否则不修改并告诉我当前的实际值。”​​
  • 原子性：整个操作（比较 + 交换）由硬件（如CPU指令）保证不可中断。

2. CAS 的操作原理

  • 操作数：三个参数

    1. 内存地址（V）：要修改的共享变量地址。
    2. 预期原值（A）：认为当前内存中的值。
    3. 新值（B）：希望修改后的值。

  • 执行流程：

    if (*V == A) {
    *V = B;
    return true;  // 修改成功
} else {
    return false; // 修改失败
}

1. 锁升级流程

回答结构：

  1. 偏向锁（优先尝试）：

    • 目的：减少无竞争时的锁开销。

      机制：

      • 首次加锁时，通过CAS将对象头的Mark Word设置为偏向线程ID，进入偏向模式。
      • 后续同一线程重入时，只需检查线程ID是否匹配，无需额外操作。

  2. 轻量级锁（竞争轻微时）：

    • 触发条件：偏向锁被其他线程访问（撤销偏向）。

      机制：

      • 线程在栈帧中创建锁记录（Lock Record），通过CAS将对象头替换为指向锁记录的指针。

  3. 重量级锁（竞争激烈时）：

    • 触发条件：自旋失败或竞争线程数超过阈值。

      机制：

      • 锁膨胀为重量级锁，对象头指向操作系统级的Monitor（ObjectMonitor）。
      • 未抢到锁的线程进入EntryList阻塞队列，等待持有线程释放后唤醒（通过waitSet处理wait/notify）。

偏向锁

轻量级锁在没有竞争时，每次重入仍然需要执行CAS操作。

Java6 中引入了偏向锁来做进一步优化：只有第一次使用CAS将线程ID设置到对象的Mark Word头，之后发现这恶线程ID是自己的就表示没有竞争，不用重新CAS，以后只要不发生竞争，这个对象就归线程锁所有。

偏向锁是在对象创建时就可以选择开启或者不开启的（默认开启），其对象头最后三位是101表示开启了偏向锁，当给对象加上synchronized的时候，除了最后的101外，还会让对象头指向当前线程的id

说了这么多，偏向锁（Biased Locking）主要用于优化单线程重复获取锁的场景，减少无竞争情况下的同步开销。

  1. 消除无竞争下的同步成本 当锁被同一线程多次获取时，传统的轻量级锁（如CAS操作）仍需执行原子指令，而偏向锁通过记录线程ID，使得后续获取锁无需任何同步操作（如CAS），直接检查线程ID即可，从而降低开销。
  2. 减少锁竞争场景的系统调用 重量级锁（如操作系统层面的互斥量）在竞争时会导致线程挂起和上下文切换，而偏向锁通过避免锁升级到重量级状态，减少了此类性能损耗。

轻量级锁的作用

轻量级锁（Lightweight Lock）是Java中针对低竞争场景设计的锁优化机制，主要用于减少多线程交替执行但未真正并发竞争时的同步开销。以下是其核心解决的问题、实现原理及适用场景：

解决的问题

  1. 降低线程阻塞的成本 重量级锁依赖操作系统的互斥量（Mutex），线程竞争锁失败时会触发上下文切换和内核态阻塞，而轻量级锁通过​​自旋（CAS操作）​​避免线程挂起，减少用户态与内核态的切换开销。
  2. 优化短时锁竞争场景 当多个线程​​交替持有锁但未发生并发竞争​​时（例如线程A释放锁后，线程B立即获取），轻量级锁通过CAS操作快速完成锁的获取与释放，避免升级为重量级锁。
  3. 减少内存占用 轻量级锁通过​​线程栈的锁记录（Lock Record）​​替代重量级锁的复杂结构（如Monitor对象），降低内存开销。

实现原理

  1. 对象头与锁标记 每个对象头（Object Header）中的Mark Word会记录锁状态。轻量级锁状态下，Mark Word存储指向​​线程栈中锁记录（Lock Record）的指针​​。

  2. 锁获取流程

    • 线程尝试获取锁时，JVM会复制对象头的Mark Word到线程栈的Lock Record中（称为Displaced Mark Word）。

    • 使用

      CAS操作

      将对象头的Mark Word替换为指向Lock Record的指针：

      • 若CAS成功，锁被当前线程获取。
      • 若CAS失败，说明存在竞争，轻量级锁会膨胀为重量级锁。

  3. 锁释放流程

    • 释放锁时，通过CAS将Displaced Mark Word还原到对象头：
      • 若还原成功，锁释放完成。
      • 若还原失败（其他线程尝试获取锁导致锁膨胀），需进入重量级锁的释放流程。

    • 创建并关联Monitor对象（ObjectMonitor），对象头指向该Monitor。
    • 线程A释放锁，进入Monitor的等待队列（WAITING状态）。
  2. 其他线程竞争锁：
    • 后续线程直接通过Monitor的阻塞队列（Entry Set）竞争锁，绕过了偏向锁和轻量级锁阶段。

Wait和Sleep的区别

  1. sleep是Thread方法，而wait是Object方法
  2. sleep不必和synchronized配合使用，wait必须配合使用，因为sleep是线程的一个等待方法，和锁的关系不大，而wait的执行会让锁对象发生锁膨胀到重量级锁，然后把wait的线程放到WaitSet中，与锁的关系较大。并且wait能够被唤醒
  3. sleep在睡眠的同时，不会释放对象锁，但wait在等待的时候会释放对象锁

  • 使用sleep()：当线程需要暂停执行一段时间，且无需释放锁或与其他线程协调。

  • 使用wait()：当线程需要释放锁并等待某个条件满足，且依赖其他线程的通知或超时机制。

保护性暂停（Guarded Suspension）设计模式详解

1. 定义与核心思想

保护性暂停是一种多线程协作设计模式，其核心思想是：当线程需要访问某个共享资源或等待某个条件时，若条件尚未满足，则线程主动挂起（暂停执行），直到条件被其他线程满足后再恢复执行。该模式通过条件判断和线程挂起/唤醒机制，确保线程在正确时机安全地操作共享数据。

2. 模式结构

  • Guarded Object（受保护对象）：包含需要被保护的共享资源或状态，提供访问和修改方法。
  • 条件变量（Condition）：决定线程是否应该挂起或恢复的布尔条件。
  • 锁（Lock）：通过同步机制（如synchronized或显式锁）确保条件检查和状态修改的原子性。

3. 代码实现示例

以下是一个典型的生产者-消费者场景示例，使用保护性暂停模式协调线程操作：

import java.util.LinkedList;
import java.util.Queue;

public class GuardedSuspensionExample {
    private final Queue<Integer> buffer = new LinkedList<>();
    private final int maxSize = 10;

    // 生产者方法：向缓冲区添加数据，若满则等待
    public synchronized void produce(int value) throws InterruptedException {
        while (buffer.size() == maxSize) {  // 条件检查必须在循环中
            wait();  // 条件不满足，挂起当前线程
        }
        buffer.add(value);
        notifyAll();  // 唤醒所有等待线程
    }

    // 消费者方法：从缓冲区取出数据，若空则等待
    public synchronized int consume() throws InterruptedException {
        while (buffer.isEmpty()) {  // 条件检查必须在循环中
            wait();  // 条件不满足，挂起当前线程
        }
        int value = buffer.poll();
        notifyAll();  // 唤醒所有等待线程
        return value;
    }

    public static void main(String[] args) {
        GuardedSuspensionExample example = new GuardedSuspensionExample();

        // 生产者线程
        Thread producer = new Thread(() -> {
            try {
                for (int i = 0; i < 20; i++) {
                    example.produce(i);
                    System.out.println("Produced: " + i);
                    Thread.sleep(100);
                }
            } catch (InterruptedException e) {
                Thread.currentThread().interrupt();
            }
        });

        // 消费者线程
        Thread consumer = new Thread(() -> {
            try {
                for (int i = 0; i < 20; i++) {
                    int value = example.consume();
                    System.out.println("Consumed: " + value);
                    Thread.sleep(150);
                }
            } catch (InterruptedException e) {
                Thread.currentThread().interrupt();
            }
        });

        producer.start();
        consumer.start();
    }
}

代码解析：

  • 条件检查循环：while (buffer.size() == maxSize)和while (buffer.isEmpty())确保在虚假唤醒时重新检查条件。
  • 挂起与唤醒：使用wait()挂起线程，notifyAll()唤醒所有等待线程，确保条件满足后能及时恢复执行。
  • 同步机制：通过synchronized关键字保证对缓冲区的操作原子性。

  • 资源池管理：线程等待资源池中的可用资源（如数据库连接）。
  • 异步操作同步：等待某个异步操作完成后再继续执行后续逻辑。

与wait notify相比：

  • wait，notify和notifyAll 必须配合Object Monitor一起使用，而unpark不必
  • park ，unpark是以线程为单位来阻塞和唤醒线程，而notify只能随机唤醒一个等待线程，没有unpark精确。
  • park & unpark可以先unpark，而wait & notify不能先notify

park/unpark（来自LockSupport类）与wait/notify（来自Object类）是Java中两种不同的线程同步机制，主要区别如下：

1. 锁的依赖

  • wait/notify：必须在synchronized代码块或方法中使用，调用前需获取对象锁。调用wait()会释放锁，而notify()不会释放锁，需退出同步块后才释放。
  • park/unpark：无需锁，直接通过LockSupport的静态方法控制线程阻塞与唤醒，灵活性更高。

2. 调用顺序

  • wait/notify：若notify()先于wait()调用，通知会丢失，线程可能永久等待。
  • park/unpark：unpark()可先于park()调用，此时park()会立即返回，不会阻塞。

3. 线程指定

  • notify：随机唤醒一个等待线程，无法指定目标；notifyAll()唤醒所有等待线程。
  • unpark：可精确唤醒指定线程（通过Thread参数）。

4. 中断处理

  • wait：被中断时抛出InterruptedException。
  • park：被中断时直接返回，不抛异常，需通过Thread.interrupted()检查中断状态。

5. 许可机制

  • park/unpark：基于“许可”模型。unpark()为线程发放一个许可（最多累积一个），park()消耗许可（有许可时不阻塞）。
  • wait/notify：无类似机制，依赖锁和条件判断。

6. 锁释放行为

  • wait()：阻塞时释放锁，允许其他线程进入同步块。
  • park()：不会释放任何锁，若在同步块中调用可能导致死锁。

适用场景

  • wait/notify：适合经典的生产者-消费者模型，需与synchronized配合使用。
  • park/unpark：适合实现底层同步工具（如AQS），需精细控制线程阻塞与唤醒，避免锁竞争。

1. 更灵活的锁控制

  • 尝试非阻塞获取锁（避免死锁）： 使用tryLock()方法，线程可以​​立即返回​​是否成功获取锁，无需无限等待。

        // 操作临界资源
    } finally {
        lock.unlock();
    }
} else {
    // 执行其他逻辑（避免阻塞）
}

  • 超时获取锁（防止无限等待）： 使用tryLock(long timeout, TimeUnit unit)，在指定时间内获取锁，超时后返回失败。

    if (lock.tryLock(3, TimeUnit.SECONDS)) { 
    // 3秒内获取到锁
}

  • 可中断的锁获取（响应中断）： 使用lockInterruptibly()，允许在等待锁的过程中被其他线程中断。

  • 公平锁：按照线程请求锁的顺序分配锁（减少饥饿，但吞吐量低）。

答：无并发问题。 🔍 原理：枚举实例在类加载阶段由 JVM ​​静态初始化​​，线程安全（饿汉式）。

答：不能。 🔍 原因：Java 反射禁止通过 Constructor#newInstance() 创建枚举对象，执行时会抛出 IllegalArgumentException。

答：饿汉式。 🔍 原因：枚举实例在类加载时初始化，而非首次调用 getInstance() 时（若需要懒加载，需结合静态内部类）。

问题1：volatile 的作用

答： volatile 关键字在此处解决两个问题：

  1. 可见性：确保多线程环境下，一个线程修改 INSTANCE 后，其他线程能立即看到最新值。
  2. 禁止指令重排序：防止 new Singleton() 的初始化操作被 JVM 重排序，导致其他线程获取到未完全初始化的对象。 🔍 原理：INSTANCE = new Singleton() 的实际操作分为三步：
  • 分配内存空间
  • 初始化对象
  • 将引用赋值给变量 若不加 volatile，步骤可能被重排为 1→3→2，导致其他线程访问到未初始化的对象。

为什么无锁的 CAS 也能保证线程安全？

你的疑问非常关键！问题的核心在于 CAS 操作本身的原子性。虽然代码中没有显式加锁，但 CAS 的“比较-交换”操作在底层是通过 CPU 硬件指令（如 cmpxchg）实现的，整个过程是 不可分割的原子操作。即使多个线程同时调用 compareAndSet，也 只有其中一个线程能成功。具体原理如下：

1. 原子性保证：硬件级“锁”

  • 硬件支持：CPU 在执行 CAS 指令时，会锁定总线（或缓存行），确保操作期间其他线程无法干扰。
  • 原子操作：比较内存值（prev）和更新为新值（next）这两个动作 在一条指令中完成，不会被线程切换打断。

注意，CAS操作必须借助Volatile才能读取到共享变量的最新值来实现【比较并交换】的效果，其实也很好理解，Volatile本来就是为了可见性设置的。

为什么无锁效率高

  • 无锁情况下，即使重复失败，线程始终在高速运行，没有停歇，而synchronized会让线程在没有获得锁的情况下，发生上下文切换，进入阻塞。

ABA问题

ABA 问题简介

问题：CAS 操作中，共享变量的值从 A → B → A，虽最终值未变，但中间可能被其他线程修改过，导致逻辑错误。

示例场景

  1. 初始状态：账户余额 100 元。

  2. 线程1：读取余额为 100，计划扣减 50（预期结果 50）。

线程2

：抢先完成两次操作：

  • 第一次扣减 50 → 余额变为 50。
  • 第二次充值 50 → 余额恢复 100。
  1. 线程1：执行 CAS 检查，发现余额仍为 100，误认为未被修改，扣减成功（最终余额 50）。

结果：正确结果应为 100 - 50 = 50，但中间线程2的修改被忽略，可能引发业务异常（如重复扣款）。

解决方案

为变量附加 版本号（AtomicStampedReference）或 标记位（AtomicMarkableReference），CAS 时同时检查值和标记，避免误判。

原子数组详解

原子数组是 Java 并发包（J.U.C）提供的线程安全数组工具类，用于在多线程环境下对数组元素进行原子操作。其核心原理是通过 volatile 内存语义 + CAS 操作 保证元素更新的原子性。以下是具体用法和场景分析：

1. 原子数组分类

类名 作用 底层存储
AtomicIntegerArray 对 int[] 元素的原子操作 volatile int[]
AtomicLongArray 对 long[] 元素的原子操作 volatile long[]
AtomicReferenceArray 对对象引用数组的原子操作 volatile Object[]

2. 核心方法（以 AtomicIntegerArray 为例）

  • get(int i)：原子读取索引 i 处的值。
  • set(int i, int newValue)：原子设置索引 i 处的值。
  • compareAndSet(int i, int expect, int update)：若索引 i 的当前值等于 expect，则更新为 update。
  • getAndAdd(int i, int delta)：原子增加索引 i 处的值，返回旧值。
  • incrementAndGet(int i)：原子自增索引 i 处的值，返回新值。

3. 代码示例：多线程累加数组元素 

import java.util.concurrent.atomic.AtomicIntegerArray;

public class AtomicArrayDemo {
    static AtomicIntegerArray array = new AtomicIntegerArray(10);  // 初始全为 0

    public static void main(String[] args) throws InterruptedException {
        Runnable task = () -> {
            for (int i = 0; i < array.length(); i++) {
                array.getAndAdd(i, 1);  // 每个元素自增 1
            }
        };

        // 启动 1000 个线程并发操作
        Thread[] threads = new Thread[1000];
        for (int i = 0; i < threads.length; i++) {
            threads[i] = new Thread(task);
            threads[i].start();
        }
        for (Thread t : threads) t.join();

        // 输出结果（每个元素应为 1000）
        System.out.println("Final Array: " + array);
    }
}

输出：

Final Array: [1000, 1000, 1000, 1000, 1000, 1000, 1000, 1000, 1000, 1000]

说明：

  • 即使 1000 个线程并发修改数组元素，结果依然正确。
  • 每个元素的更新通过 CAS 保证原子性，避免线程竞争导致的数据不一致。

原子累加器

LongAdder 是什么？

LongAdder 是 Java 8 引入的高性能并发累加器类（位于 java.util.concurrent.atomic 包），专门用于高并发场景下的计数器、统计等需要频繁累加操作的场景。它通过 分段锁（Cell 分散） 策略，显著优化了多线程竞争下的吞吐量，是传统 AtomicLong 的增强替代方案。

1. 解决 AtomicLong 的性能瓶颈

  • AtomicLong 的问题： 通过 CAS 保证线程安全，但在高并发时，大量线程竞争修改同一个变量会导致频繁的 CAS 失败重试，性能急剧下降。
  • LongAdder 的优化： 将累加值分散到多个 Cell 单元中，线程优先修改各自独立的 Cell，最终结果取所有 Cell 的求和值，​​极大减少竞争​​。

2. 性能对比（高并发场景）

操作 AtomicLong LongAdder
单线程累加 快（无竞争） 稍慢（有额外开销）
高并发累加 慢（竞争激烈） 快（竞争分散）

核心原理

  1. 分段累加（Cell 数组）：

    • 初始时，所有线程累加到一个 base 值。
    • 当检测到竞争（CAS 失败），动态创建 Cell[] 数组，将线程的累加操作分散到不同的 Cell 单元。
    • 线程优先操作其哈希值对应的 Cell，减少冲突。

  2. 最终结果计算

    public long sum() {
    long sum = base;
    for (Cell cell : cells) {
        if (cell != null) sum += cell.value;
    }
    return sum;
}

适用场景

  • 高并发计数器：如统计请求量、在线人数、点击量等。
  • 实时性要求较低的场景：sum() 方法返回的是最终一致性结果（非实时精确值）。
  • 替代 AtomicLong：当 AtomicLong 在高并发下成为性能瓶颈时。

代码示例 

import java.util.concurrent.atomic.LongAdder;

public class LongAdderDemo {
    public static void main(String[] args) {
        LongAdder adder = new LongAdder();
        
        // 启动 100 个线程并发累加
        for (int i = 0; i < 100; i++) {
            new Thread(() -> {
                for (int j = 0; j < 1000; j++) {
                    adder.increment();  // 自增
                }
            }).start();
        }
        
        // 等待所有线程完成
        while (Thread.activeCount() > 2) Thread.yield();
        
        System.out.println("Total: " + adder.sum());  // 输出 100000
    }
}

享元模式

Flyweight pattern 当需要重用数量有限的同一类对象时

1. 享元模式的核心思想

享元模式是一种结构型设计模式，旨在通过共享细粒度对象来减少内存占用和对象创建开销。其核心原则是：

  • 区分内部状态与外部状态：

    • 内部状态：对象中可共享的部分（例如字符内容）。
    • 外部状态：对象中依赖外部传入的参数（例如位置、样式）。

  • 复用不可变对象：若多个场景需要同一内部状态，共享一个实例而非重复创建。

经典应用场景

  • 文本编辑器中的字符渲染（共享字形数据）。
  • 游戏中的粒子系统（共享粒子纹理、模型）。
  • Java 包装类的缓存（如 Integer.valueOf(-128~127)）。

2. 享元模式的结构

角色 作用
Flyweight 定义共享对象的接口（如字符的渲染方法）。
ConcreteFlyweight 实现具体共享对象（如字符 'A' 的实例）。
FlyweightFactory 管理共享对象的工厂（缓存和复用逻辑）。
Client 使用享元对象，传递外部状态（如字符位置）。

3. 享元模式 vs. 保护性拷贝

通过对比 String 类的保护性拷贝与享元模式，理解两者的设计差异：

特性 保护性拷贝（如 String） 享元模式
目标 确保对象不可变性和线程安全 减少内存开销，共享重复数据
内存策略 为每个对象创建独立副本 共享内部状态，仅存储一份副本
适用场景 高安全性需求（如字符串、日期） 高频重复数据（如字符、配置参数）
性能影响 牺牲内存换取安全 牺牲部分灵活性换取内存优化

案例分析

  • String 类的设计：

    • 使用保护性拷贝，确保每个 String 的 char[] 独立，避免外部篡改。
    • 未采用享元模式：即使两个字符串内容相同，仍会创建独立对象（如 new String("abc")）。
    • 例外优化：字符串常量池（String.intern()）是享元模式的体现，但非常量字符串不共享。

  • Java 包装类的缓存：

    Integer a = Integer.valueOf(127);  // 从缓存获取
Integer b = Integer.valueOf(127);  // 共享同一对象
System.out.println(a == b);        // 输出 true

4. 享元模式的实现示例

场景描述

实现一个文本编辑器，渲染大量重复字符（如字母 A），通过共享字形数据优化内存。

代码实现 

// 1. Flyweight 接口
interface Glyph {
    void render(int x, int y);
}

// 2. ConcreteFlyweight（共享的字形数据）
class CharacterGlyph implements Glyph {
    private final char character;  // 内部状态（共享）

    public CharacterGlyph(char c) {
        this.character = c;
    }

    @Override
    public void render(int x, int y) {
        System.out.printf("Render '%c' at (%d, %d)\n", character, x, y);
    }
}

// 3. FlyweightFactory（缓存字形对象）
class GlyphFactory {
    private static final Map<Character, Glyph> pool = new HashMap<>();

    public static Glyph getGlyph(char c) {
        if (!pool.containsKey(c)) {
            pool.put(c, new CharacterGlyph(c));  // 首次创建后缓存
        }
        return pool.get(c);
    }
}

// 4. Client 使用
public class TextEditor {
    public static void main(String[] args) {
        // 渲染 1000 个字母 'A'，共享同一个 Glyph 对象
        Glyph a = GlyphFactory.getGlyph('A');
        for (int i = 0; i < 1000; i++) {
            a.render(i * 10, 0);  // 外部状态（位置）由客户端传递
        }
    }
}

final原理

加了final，会把较小值放到栈中，较大对象放到常量池中，访问会更快。

final 写屏障的核心作用：

  • 禁止指令重排序：确保 final 字段的赋值操作不会被重排序到构造函数之外。
  • 可见性保证：通过写屏障刷新 CPU 缓存，使其他线程读取到正确的初始值（防止读到默认值 0）。

inal 的不可变性由 多层级机制 共同保障：

  1. 编译期：禁止重复赋值。
  2. 字节码与 JVM：通过写屏障确保初始值的可见性，并禁止指令重排序。
  3. 内存模型：对象发布时保证所有 final 字段已完成初始化。

这些机制使得 final 变量在 多线程环境下无需额外同步即可安全访问，是构建线程安全对象（如 String）的基石。

class ThreadPool {
    // 阻塞队列
    private BlockingQueue<Runnable> taskQueue;

    // 线程集合
    private HashSet<Worker> workers = new HashSet<>();

    // 最大容量
    private int coreSize;

    // 获取任务超时时间
    private long timeout;

    private TimeUnit timeUnit;
    // 执行任务
    public void execute(Runnable task) {
        //如果任务数没有超过size，就添加到workers中去执行，否则添加到阻塞队列
        synchronized (workers) {
            if(workers.size() < coreSize) {
                Worker worker = new Worker(task);
                workers.add(worker);
                worker.start();
            } else {
                taskQueue.put(task);
            }
        }
    }
    public ThreadPool(TimeUnit timeUnit, long timeout, int coreSize,  int queueCapacity) {
        this.timeUnit = timeUnit;
        this.timeout = timeout;
        this.coreSize = coreSize;
        this.taskQueue = new BlockingQueue<>(queueCapacity);
    }

    class Worker extends Thread {
        private Runnable task;

        public Worker(Runnable task) {
            this.task = task;
        }
        @Override
        public void run() {
            // when task is not null , execute the task
            // if the task is null,to get task from BlockingQueue
            while(task != null || (task = taskQueue.take()) != null) {
                try {
                    task.run();
                } catch (Exception e) {
                    throw new RuntimeException(e);
                } finally {
                    task = null;
                }
            }
            synchronized (workers) {
                workers.remove(this);
            }
        }
    }
}

class BlockingQueue<T> {
    // 1. 任务队列
    private Deque<T> queue = new ArrayDeque<T>();

    // 2. 锁
    private ReentrantLock lock = new ReentrantLock();

    // 3. 生产者条件变量
    private Condition fullWaitSet = lock.newCondition();

    // 4. 消费者条件变量
    private Condition emptyWaitSet = lock.newCondition();

    // 5. 容量
    private int capacity;

    public BlockingQueue(int capacity) {
        this.capacity = capacity;
    }

    // 带超时的阻塞获取
    public T poll(long timeout, TimeUnit unit) throws InterruptedException {
        lock.lock();
        try {
            // 将 timeout 同一转换为 纳秒
            long nanos = unit.toNanos(timeout);
            while (queue.isEmpty()) {
                try {
                    if(nanos <= 0) {
                        return null;
                    }
                    nanos = emptyWaitSet.awaitNanos(nanos);
                } catch (InterruptedException e) {
                    throw new RuntimeException(e);
                }
            }
            T t = queue.removeFirst();
            fullWaitSet.signal();
            return t;
        } finally {
            lock.unlock();
        }
    }

    // 阻塞获取
    public T take() {
        lock.lock();
        try {
            while (queue.isEmpty()) {
                try {
                    emptyWaitSet.await();
                } catch (InterruptedException e) {
                    throw new RuntimeException(e);
                }
            }
            T t = queue.removeFirst();
            fullWaitSet.signal();
            return t;
        } finally {
            lock.unlock();
        }
    }

    // 阻塞添加
    public void put(T element) {
        try {
            lock.lock();
            while (queue.size() == capacity) {
                try {
                    fullWaitSet.await();
                } catch (InterruptedException e) {
                    throw new RuntimeException(e);
                }
            }
            queue.addLast(element);
            emptyWaitSet.signal();
        } finally {
            lock.unlock();
        }
    }

    // 获取大小
    public int size() {
        lock.lock();
        try {
            return queue.size();
        } finally {
            lock.unlock();
        }
    }

}

AQS

全称是AbstractQueuedSynchronizer，是阻塞式和相关的同步器工具的框架。

特点：

  • 用state属性来表示资源的状态，子类需要定义如何维护这个状态，控制如何获取锁和释放锁
    • getState 获取state状态
    • setState 设置state状态
    • compareAndSetState 乐观锁机制设置state状态
    • 独占模式是只有一个线程能够访问资源，而共享模式可以允许多个线程访问资源，
  • 提供了基于FIFO的等待队列，类似于Monitor的EntryList
  • 条件变量来实现等待，唤醒机制，支持多个条件变量，类似于Monitor的WaitSet

AQS（AbstractQueuedSynchronizer）一句话总结： Java并发包中​​同步器的核心框架​​，用来快速实现锁、信号量等同步工具（如ReentrantLock/Semaphore等）的底层基石。

为什么需要AQS？

  1. 统一模板：解决线程同步问题（如锁竞争、资源控制）的通用实现模板，避免重复造轮子
  2. 高效排队：通过CLH队列管理等待线程（类似「排队等餐」的取号机），减少CPU空转
  3. 状态管理：用volatile int state原子变量统一表示资源状态（如锁的持有次数、信号量剩余数量）

数据丢失场景分析

  • Thread-1 遍历链表，处理节点

    A
    • 计算 A 的新位置（假设分配到低位链表）。
    • 将 A 挂载到新数组的低位链表头部。

  • 挂起前状态：Thread-1 的 next 指针指向 B。

步骤 2：Thread-2 完整迁移链表

  • Thread-2 独立完成链表

    A → B → C

    的迁移：

    • 拆分并挂载 A、B、C 到新数组。
    • 新数组的低位链表为 A → B → C → null。

步骤 3：Thread-1 恢复执行

  • Thread-1 继续处理

    B

    和

    C

    ，但此时：

    • 旧链表的 B 和 C 已被 Thread-2 迁移到新数组。
    • Thread-1 的 next 指针仍指向 B（基于旧链表状态）。

  • 操作结果：

    • Thread-1 将 B 和 C 重复挂载 到新数组的低位链表头部。

大总结-Java锁机制

广义上看，可以将锁分为乐观锁和悲观锁，体现了看待线程同步的不同角度。

对同一个数据的并发操作，悲观锁认为自己在使用数据的时候一定有别的线程来修改数据，因此在获取数据的时候会先加锁，确保数据不会被别的线程修改。Java中，synchronized关键字和Lock的实现类都是悲观锁。

而乐观锁认为自己在使用数据时不会有别的线程修改数据，所以不会添加锁，只是在更新数据的时候去判断之前有没有别的线程更新了这个数据，如果这个数据没有被更新，当前线程将自己修改的数据成功写入。例如CAS操作

  • 悲观锁适合写操作多的场景，先加锁可以保证写操作时数据正确。
  • 乐观锁适合读操作多的场景，不加锁的特点能够使其读操作的性能大幅提升。

而CAS之所以能够实现乐观锁，保证其更新时的原子性，在于其最底层是CPU指令来保证其原子性的。

当然，CAS也有其对应的问题：

  1. ABA问题

    。CAS需要在操作值的时候检查内存值是否发生变化，没有发生变化才会更新内存值。但是如果内存值原来是A，后来变成了B，然后又变成了A，那么CAS进行检查时会发现值没有发生变化，但是实际上是有变化的。ABA问题的解决思路就是在变量前面添加版本号，每次变量更新的时候都把版本号加一，这样变化过程就从“A－B－A”变成了“1A－2B－3A”。

    • JDK从1.5开始提供了AtomicStampedReference类来解决ABA问题，具体操作封装在compareAndSet()中。compareAndSet()首先检查当前引用和当前标志与预期引用和预期标志是否相等，如果都相等，则以原子方式将引用值和标志的值设置为给定的更新值。

  2. 循环时间长开销大。CAS操作如果长时间不成功，会导致其一直自旋，给CPU带来非常大的开销。

  3. 只能保证一个共享变量的原子操作

    。对一个共享变量执行操作时，CAS能够保证原子操作，但是对多个共享变量操作时，CAS是无法保证操作的原子性的。

    • Java从1.5开始JDK提供了AtomicReference类来保证引用对象之间的原子性，可以把多个变量放在一个对象里来进行CAS操作。

从CAS操作的延申，又可以引出自旋锁，例如AtomicInteger进行自增操作就是一个自旋操作。

再回到悲观锁，悲观锁的重点自然是synchronized，而synchronized实现线程同步是通过Monitor锁对象，每一个被锁住的对象都会和一个monitor关联，同时monitor中有一个Owner字段存放拥有该锁的线程的唯一标识，表示该锁被这个线程占用。

synchronized通过Monitor实现线程同步，Monitor是依赖于底层的操作系统的Mutex Lock来实现的线程同步。

阻塞或唤醒一个Java线程需要操作系统切换CPU状态来完成，这种状态转换需要耗费处理器时间。如果同步代码块中的内容过于简单，状态转换消耗的时间有可能比用户代码执行的时间还要长。这种方式就是synchronized最初实现同步的方式，这就是JDK 6之前synchronized效率低的原因。这种依赖于操作系统Mutex Lock所实现的锁我们称之为“重量级锁”，JDK 6中为了减少获得锁和释放锁带来的性能消耗，引入了“偏向锁”和“轻量级锁”。

所以目前锁一共有4种状态，级别从低到高依次是：无锁、偏向锁、轻量级锁和重量级锁。锁状态只能升级不能降级。

无锁

无锁没有对资源进行锁定，所有的线程都能访问并修改同一个资源，但同时只有一个线程能修改成功。

无锁的特点就是修改操作在循环内进行，线程会不断的尝试修改共享资源。如果没有冲突就修改成功并退出，否则就会继续循环尝试。如果有多个线程修改同一个值，必定会有一个线程能修改成功，而其他修改失败的线程会不断重试直到修改成功。上面我们介绍的CAS原理及应用即是无锁的实现。无锁无法全面代替有锁，但无锁在某些场合下的性能是非常高的。

偏向锁

偏向锁是指一段同步代码一直被一个线程所访问，那么该线程会自动获取锁，降低获取锁的代价。

在大多数情况下，锁总是由同一线程多次获得，不存在多线程竞争，所以出现了偏向锁。其目标就是在只有一个线程执行同步代码块时能够提高性能。

当一个线程访问同步代码块并获取锁时，会在Mark Word里存储锁偏向的线程ID。在线程进入和退出同步块时不再通过CAS操作来加锁和解锁，而是检测Mark Word里是否存储着指向当前线程的偏向锁。引入偏向锁是为了在无多线程竞争的情况下尽量减少不必要的轻量级锁执行路径，因为轻量级锁的获取及释放依赖多次CAS原子指令，而偏向锁只需要在置换ThreadID的时候依赖一次CAS原子指令即可。

轻量级锁

当锁是偏向锁的时候，被另外的线程所访问，偏向锁就会升级为轻量级锁，其他线程会通过自旋的形式尝试获取锁，不会阻塞，从而提高性能。

在代码进入同步代码块的时候，虚拟机会在当前线程的栈帧中的建立一个名为（Lock Record）的空间，用于存放锁对象目前的Mark word的拷贝，然后将对象头的Mark Word更新为指向Lock Record的指针，并将Lock Record里的owner指针指向对象。

如果这个更新动作成功了，那么这个线程就拥有了该对象的锁，并且对象Mark Word的锁标志位设置为“00”，表示此对象处于轻量级锁定状态。

如果轻量级锁的更新操作失败了，虚拟机首先会检查对象的Mark Word是否指向当前线程的栈帧，如果是就说明当前线程已经拥有了这个对象的锁，那就可以直接进入同步块继续执行，否则说明多个线程竞争锁。

若当前只有一个等待线程，则该线程通过自旋进行等待。但是当自旋超过一定的次数，或者一个线程在持有锁，一个在自旋，又有第三个来访时，轻量级锁升级为重量级锁。

重量级锁

即轻量级锁多个竞争时，为锁对象申请Monitor对象，并将对象头指向Monitor对象，竞争线程进入EntryList阻塞等待。

以上是从悲观锁的角度聊到了synchronized，降到了synchronized在JDK6的优化。当然，悲观锁不止synchronized，ReentrantLock也是悲观锁的一个主要实现。

谈到ReentrantLock，就不得不说AQS，AQS主要是两个概念，state管理和CLH队列，而state管理通过CAS来保证原子操作变更安全，通过unsafe类调用本地方法。

ReentrantLock底层调用的正是AQS封装好的一些方法，要注意的是，AQS使用CAS（乐观锁机制）修改state，但是表现到上层的ReentrantLcok，却是悲观锁。

因为ReentrantLock的lock方法是使用CAS快速尝试获取锁，但是拿不到锁的话会进入CLH队列，阻塞，所以乐不乐观关键看线程有没有被阻塞，有没有涉及线程上下文的切换。底层的AQS在修改state状态时使用了乐观锁机制，但是CLH阻塞队列使用的是悲观锁机制。

当然了，ReentrantLock还有一个重要的机制就是公平锁和非公平锁

公平锁是指多个线程按照申请锁的顺序来获取锁，线程直接进入队列中排队，队列中的第一个线程才能获得锁。公平锁的优点是等待锁的线程不会饿死。缺点是整体吞吐效率相对非公平锁要低，等待队列中除第一个线程以外的所有线程都会阻塞，CPU唤醒阻塞线程的开销比非公平锁大。

非公平锁是多个线程加锁时直接尝试获取锁，获取不到才会到等待队列的队尾等待。但如果此时锁刚好可用，那么这个线程可以无需阻塞直接获取到锁，所以非公平锁有可能出现后申请锁的线程先获取锁的场景。非公平锁的优点是可以减少唤起线程的开销，整体的吞吐效率高，因为线程有几率不阻塞直接获得锁，CPU不必唤醒所有线程。缺点是处于等待队列中的线程可能会饿死，或者等很久才会获得锁。

我们前面提到AQS机制下线程竞争失败时，我们采用CLH队列来存放阻塞队列，这个队列本质是用一个链表维护的，在公平锁机制下，一个线程正是由其前面的一个线程节点来唤醒的。在尝试获取同步状态时，公平锁会先判断当前线程是否是队列中的第一个，如果不是则获取不到，而非公平锁不去判断这一条件，直接进行竞争获取。

synchronized并没有公平不公平这一说法，因为其是随机唤醒阻塞的队列。与reentrantLock相比，二者都实现了可重入锁机制

维度 synchronized ReentrantLock

独享锁与共享锁

6. 独享锁 VS 共享锁

独享锁和共享锁同样是一种概念。我们先介绍一下具体的概念，然后通过ReentrantLock和ReentrantReadWriteLock的源码来介绍独享锁和共享锁。

独享锁也叫排他锁，是指该锁一次只能被一个线程所持有。如果线程T对数据A加上排它锁后，则其他线程不能再对A加任何类型的锁。获得排它锁的线程即能读数据又能修改数据。JDK中的synchronized和JUC中Lock的实现类就是互斥锁。

共享锁是指该锁可被多个线程所持有。如果线程T对数据A加上共享锁后，则其他线程只能对A再加共享锁，不能加排它锁。获得共享锁的线程只能读数据，不能修改数据。

独享锁与共享锁也是通过AQS来实现的，通过实现不同的方法，来实现独享或者共享。所以显然读锁使用的就是共享锁，而写锁使用独享锁

ReentrantReadWriteLock有两把锁：ReadLock和WriteLock，由词知意，一个读锁一个写锁，合称“读写锁”。再进一步观察可以发现ReadLock和WriteLock是靠内部类Sync实现的锁。Sync是AQS的一个子类，这种结构在CountDownLatch、ReentrantLock、Semaphore里面也都存在。

在ReentrantReadWriteLock里面，读锁和写锁的锁主体都是Sync，但读锁和写锁的加锁方式不一样。读锁是共享锁，写锁是独享锁。读锁的共享锁可保证并发读非常高效，而读写、写读、写写的过程互斥，因为读锁和写锁是分离的。所以ReentrantReadWriteLock的并发性相比一般的互斥锁有了很大提升。

在独享锁中这个值通常是0或者1（如果是重入锁的话state值就是重入的次数），在共享锁中state就是持有锁的数量。但是在ReentrantReadWriteLock中有读、写两把锁，所以需要在一个整型变量state上分别描述读锁和写锁的数量（或者也可以叫状态）。于是将state变量“按位切割”切分成了两个部分，高16位表示读锁状态（读锁个数），低16位表示写锁状态（写锁个数）。如下图所示：
